# 稳定性修复修改内容说明（2026-03-25）

## 1. 目标与背景
本次修改针对“打开游戏并获取数据后，设备卡死并自动重启”的问题进行稳定性修复。
核心策略：降低驱动读压、增加读失败防护、减少渲染线程直连驱动读内存。

## 2. 修改范围
- `src/Android_draw/draw_Gui.cpp`
- `src/Android_draw/kerneldriver-qxqd.hpp`

## 3. 关键改动

### 3.1 读取线程限流与降载
- 新增参数：`kReadLoopSleepUs = 50000`（50ms）
- 新增参数：`kMaxEntityScanCount = 600`
- 作用：降低 `ioctl(OP_READ_MEM)` 调用频率和单轮扫描量，减少内核驱动压力。
- 位置：
  - `draw_Gui.cpp:176`
  - `draw_Gui.cpp:177`
  - `draw_Gui.cpp:422`
  - `draw_Gui.cpp:518`

### 3.2 指针链路改为安全分步读取
- 新增安全读取函数：
  - `IsLikelyPtr`
  - `ReadPtr64Safe`
  - `ReadI32Safe`
  - `ReadF32Safe`
  - `ReadU8Safe`
- 将原来类名链 5 层连读替换为逐层判空/判失败，读失败立即 `continue`。
- 作用：避免悬空地址/无效地址继续解引用导致驱动异常。
- 位置：
  - `draw_Gui.cpp:179-210`
  - `draw_Gui.cpp:452-456`

### 3.3 渲染线程去驱动直读（矩阵缓存化）
- 扩展三缓冲结构 `GameDataBuffer`，新增：
  - `float matrix[16]`
  - `bool matrix_valid`
- 读取线程预读矩阵并写入缓冲；渲染线程直接使用缓冲矩阵。
- 作用：降低渲染帧循环中的驱动访问次数。
- 位置：
  - `draw_Gui.cpp:52-57`
  - `draw_Gui.cpp:430-439`
  - `draw_Gui.cpp:620-625`
  - `draw_Gui.cpp:736-743`

### 3.4 对象数据预取缓存化
- 扩展 `DataStruct`，新增：
  - `Vector3A pos`
  - `bool has_pos`
  - `bool is_ghost_obj`
  - `uint8_t flag_aa`
- 读取线程预读坐标、有效性字段和 `0xAA` 标记；绘制逻辑优先使用缓存。
- 作用：减少绘制路径中的驱动随机读。
- 位置：
  - `draw_Gui.cpp:40-50`
  - `draw_Gui.cpp:478-493`
  - `draw_Gui.cpp:302-326`
  - `draw_Gui.cpp:653`

### 3.5 字符串与读取失败安全性增强
- 将多处 `strcpy` 改为 `snprintf`，避免潜在溢出。
- 扫描阶段 `vm_readv` 失败时跳过当前页，不继续使用脏数据。
- 位置：
  - `draw_Gui.cpp:392-394`
  - `draw_Gui.cpp:471`
  - `draw_Gui.cpp:497`
  - `draw_Gui.cpp:501`
  - `draw_Gui.cpp:504`
  - `draw_Gui.cpp:507`
  - `draw_Gui.cpp:510`

### 3.6 STB 实现宏重复定义问题处理
- 删除 `draw_Gui.cpp` 中的 `#define STB_IMAGE_IMPLEMENTATION`，保留 `src/My_Utils/stb_image.cpp` 的唯一实现。
- 作用：避免重复实现引起的不确定链接/运行时风险。

### 3.7 底层 `getPtr64` 失败返回补丁
- `driver->read` 失败时，`getPtr64` 返回 `0`。
- 作用：避免读取失败时把未定义值当有效地址继续使用。
- 位置：`kerneldriver-qxqd.hpp:370-378`

## 4. 验证结果
- 构建命令：
  - `ndk-build -C . NDK_OUT=./obj NDK_LIBS_OUT=./libs`
- 构建结果：通过，产物生成于 `jni/libs/arm64-v8a/Identity5.sh`。

## 5. 可调参数（后续压测建议）
- `kReadLoopSleepUs`：可在 `50000~100000` 之间调节。
- `kMaxEntityScanCount`：可在 `300~600` 之间调节。
- 当机型驱动稳定性较差时，优先提高 `kReadLoopSleepUs`。
